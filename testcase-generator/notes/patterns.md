# 발견한 패턴 / 안티패턴

> TC 생성 과정에서 발견한 효과적인 패턴과 피해야 할 안티패턴을 기록합니다.

---

## 효과적인 패턴 (Best Practices)

### 1. PPTX 구조 분석 패턴

**패턴명**: 계층적 슬라이드 분석
**설명**: 슬라이드 → 그룹 → 요소 순서로 분석
**장점**: Depth 구조 정확도 향상
**적용 예시**:
```python
for slide in pptx.slides:
    for group in slide.groups:
        for element in group.shapes:
            # 처리
```

### 2. TC ID 생성 패턴

**패턴명**: 컴포넌트 기반 ID
**설명**: `IT_[PREFIX]_[기능코드]_[순번]` 형식
**장점**: ID만으로 기능 파악 가능
**적용 예시**: `IT_{PREFIX}_BT_001` → {프로젝트명} Button TC #1

### 3. Expected Result 작성 패턴

**패턴명**: 다중 해시 결과문
**설명**: 각 검증 포인트를 `#`으로 구분
**장점**: 테스트 결과 판정 명확화
**적용 예시**:
```
# 저장 완료 메시지 표시
# 파일이 지정 경로에 저장됨
# 화면 상태 유지
```

## 피해야 할 안티패턴

### 0. 하드코딩된 TC 내용 (최우선 금지)

**안티패턴명**: 화면정의서 외 데이터 사용
**설명**: 키워드 매칭으로 탭명 추정, 컴포넌트별 위치 가정
**문제점**:
- "Tool 탭 화면 진입" - PPTX에 "Tool 탭"이 없음
- "최소화 버튼 = 우측 상단" - PPTX에 위치 정보 없음
**개선 방향**:
- PPTX 섹션 제목 그대로 사용
- Description에 위치가 명시된 경우만 추출
- 없는 정보는 비워두거나 "해당 화면"으로 일반화

### 1. 모호한 Expected Result

**안티패턴명**: 추상적 결과 기술
**설명**: "정상 동작함", "예상대로 작동" 등
**문제점**: 테스트 합격/불합격 판정 불가
**개선 방향**: 구체적인 상태 변화 명시

### 2. 과도한 Pre-condition

**안티패턴명**: 불필요한 전제조건 나열
**설명**: 기본 상태를 Pre-condition에 명시
**문제점**: TC 가독성 저하, 중복 정보
**개선 방향**: 특수 조건만 Pre-condition에 명시

### 3. 단일 Test Step

**안티패턴명**: 복합 동작 한 줄 기술
**설명**: 여러 동작을 한 단계에 기술
**문제점**: 실패 지점 파악 어려움
**개선 방향**: 원자적 동작 단위로 분리

**Before/After 비교**:
```
❌ Before: "1. 팝업에서 Save 클릭"
✅ After:  "1. Alignment 탭 화면 진입
           2. 우측 상단 닫기 버튼 클릭
           3. 팝업창 Save 버튼 클릭"
```

### 4. 위치 서술자 누락

**안티패턴명**: 위치 정보 없는 동작 기술
**설명**: 버튼/컴포넌트 이름만 기술하고 화면 내 위치를 명시하지 않음
**문제점**: 테스터가 UI에서 대상을 찾기 어려움
**개선 방향**: 이미지 분석 기반 위치 서술자 필수 포함

**Before/After 비교**:
```
❌ Before: "Save 버튼 클릭"
✅ After:  "좌측 Common Tool 영역의 Save 버튼 클릭"
```

### 5. 진입 동작 생략

**안티패턴명**: 화면/탭 진입 없이 바로 동작 시작
**설명**: Step 1에서 화면 진입 없이 곧바로 클릭/입력 동작부터 시작
**문제점**: 테스터가 어느 화면에서 시작해야 하는지 알 수 없음
**개선 방향**: Step 1은 반드시 "{화면/탭명} 화면 진입" 형태

**Before/After 비교**:
```
❌ Before: "1. Save 버튼 클릭"
✅ After:  "1. 공통 Layout 화면 진입
           2. 좌측 Common Tool 영역의 Save 버튼 클릭"
```
